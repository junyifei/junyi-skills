# Build Evidence-Based IP Book

把个人经历、用户证据、内容数据和商业行动变成一份可决策、可验证、可修订的战略书。把未经检验的方向写成假设，不把好听的表达冒充证据。

## 先选择任务

选择一种模式，必要时组合：

1. **从零设计**：从访谈、资料与市场信号中建立证据账本，生成并比较 3-5 个定位方向。
2. **审核现有定位**：保留原方案，逐层检查问题出在用户想起你的时刻、人物记忆、内容承诺、能力证明、买方、产品或表达。
3. **迭代旧战略书**：先建立版本差异和新增证据，只修改失败层；明确保留、删除、降级和待验证内容。
4. **仅做定位诊断**：不强制生成整本战略书；判断定位失败层、比较候选方向，输出定位决定、证据缺口、最小修复和验证条件。

输出前先说明当前模式、已有资料、关键缺口和允许做出的决策。资料不足时继续完成可完成部分，把缺口标为“未知”，不要逼用户一次答完所有问题。

开始任何任务前，读取：

- [positioning-and-scoring.md](references/positioning-and-scoring.md)：五层定位、三类人群、变化地图、候选比较与评分。

## 核心工作流

### 1. 建立证据账本

先整理证据，再写定位。原因是战略书的每个重要结论都在消耗可信度：没有账本，个人愿望、市场观察、旧方案和真实成交很容易混成一句“已经验证”。

按 [interview-and-evidence.md](references/interview-and-evidence.md) 记录来源、时间、适用范围和授权状态，并给每条关键陈述加标签：

- `【事实】`：有原始记录、可观察结果或可靠来源。
- `【推断】`：由事实支持的合理解释，但未被直接验证。
- `【假设】`：可测试、尚未证明的主张。
- `【未知】`：做决定需要、当前没有的信息。

优先级遵循：本人或客户的真实结果与重复支付 > 单次支付或定金 > 合格咨询与具体购买问题 > 深度用户原话 > 关注、收藏、转发与完播 > 浏览与点赞 > 朋友意见或个人直觉。不同层级可以回答不同问题，不能互相冒充。

### 2. 回到个人真实资产

回答：

1. 什么经历、能力、关系、场景或观点是这个人真实拥有的？
2. 什么问题是这个人亲自走过、反复解决或持续实践的？
3. 哪些材料可以被展示为过程、产物、前后对比、案例或长期纪录？
4. 这个方向能否从真实资产中产生至少 100 个不重复选题？
5. 哪些隐私、伦理、精力、产能和生活方式边界不能被突破？

用市场研究校准需求和用户语言，不用市场热点制造一个本人无法持续扮演的人。

### 3. 先写“用户在什么情况下会想起你”

不要从“我是谁”开始。写成：

> 当哪类人遇到什么具体情况、想得到什么变化时，应该想起这个人？

要求这个情况可观察、会重复出现、对用户有后果，并能自然连接到内容和能力证明。人口属性只能帮助描述人，不能代替使用场景。

### 4. 区分五层定位

不要让一句口号同时负责传播、记忆、内容、成交和价值观。分别决定：

1. **传播入口**：陌生人为什么愿意停下来，通常是更宽的现实问题、冲突或好奇。
2. **人物标签**：别人怎样用最短、自然的一句话记住和转述这个人。
3. **内容承诺**：持续关注后会反复得到什么价值或看到什么进展。
4. **商业落点**：真正买方是谁、为什么现在要解决、购买什么结果、由什么产品承接。
5. **长期价值观**：这个人如何判断、为什么长期值得信任、哪些事即使有流量也不做。

这五层必须指向同一组变化，但宽窄可以不同。传播入口可宽于真正买方，商业落点不能为了追求大流量而被迫变宽。

### 5. 区分三类人

分别写清：

- **触达人群**：最容易理解公开内容的人，负责形成规模。
- **真正买方**：有明确问题、预算、决策权、行动意愿并适合交付的人。
- **不需要吸引的人**：会造成错误流量、错误期待、伦理风险或不适配交付的人。

不要用粉丝画像代替买方条件，也不要因为某类人会带来播放就默认应该吸引。

### 6. 建立“用户从之前到之后”的变化地图

按 [positioning-and-scoring.md](references/positioning-and-scoring.md) 分别写可观察的：

- 外部处境与行为；
- 结果与成本；
- 能力、判断或系统；
- 感受与身份；
- 必须保持不变的边界。

每个“之后”都要能被内容展示、被服务交付或被验证。避免只写“更好、更自由、更有能量”。

### 7. 生成并比较定位方向

生成 3-5 个真正不同的方向，不要只重排身份名词。每个方向至少写：用户想起你的时刻、触达人群、真正买方、变化、独有资产、五层定位、内容燃料、能力证明、产品邻近性、风险和未知。

先做“成为主定位前必须通过的基础检查”，再做加权评分。基础检查失败的方向不能靠总分补救。使用 [positioning-and-scoring.md](references/positioning-and-scoring.md)；需要确定性计算时运行：

```bash
python scripts/score_candidates.py candidates.json
```

选择一个暂定主方向，最多保留一个受控挑战方向。说明为什么选择、还缺什么证据，以及什么情况说明这条路不成立。

### 8. 设计内容、故事、能力证明和商业验证

让每个内容支柱都回到同一变化，并明确：内容任务、证据来源、稳定栏目、示例题、主要指标、与商业落点的连接。

内容可承担被发现、提供实用价值、展示人物与建立转化等不同任务，但比例必须根据账号阶段、平台、产能和数据决定，不把任何案例比例写成标准。多平台可以共用母题与真实素材，必须按各平台的用户动作分别改写开头、主体、结尾和指标，不能直接复制发布。

分开积累：

- **被理解和喜欢的信任**：真实经历、选择、价值观和持续生活证据。
- **相信你能交付结果的信任**：过程演示、案例、前后对比、可复用产物、结果与重复交付。

故事不能替代能力证明；高播放不能替代付款。建立从免费内容、最小付费测试、核心产品到更深服务的阶梯，并写清不服务什么、不承诺什么。

### 9. 设计 30 天和 90 天验证

把注意力、信任和支付分成三条证据线。每个实验预先写：假设、刺激、目标人群、指标、基线、最低样本、期限、混杂因素、通过/混合/不通过后的动作。

至少用三次同类内容测试一个注意力假设。阈值优先参考账号自己的同形式基线；没有基线时先建立基线，不随意发明行业及格线。只有真实付款、定金或其他有成本承诺才能证明支付意愿。

90 天按“启动与基线 → 验证方向 → 形成案例 → 只放大赢家”推进。根据结果只修改失败层，不因一条爆款确认定位，也不因一条低播放推翻定位。

### 10. 输出战略书并检查

按 [strategy-book-framework.md](references/strategy-book-framework.md) 输出 Markdown。先给决定，再给证据和限制，然后解释架构、用户变化、内容证明、产品、候选比较与验证路线。

运行质量检查：

```bash
python scripts/check_strategy_book.py strategy.md
```

如果资料包含不应出现在交付物中的案例人物，可额外检查案例污染：

```bash
python scripts/check_strategy_book.py strategy.md --forbid-name "<案例人物名>"
```

用户要求 Word、Google Docs 或 PDF 时，先完成并确认 Markdown 内容，再调用相应文档或 PDF 能力排版与视觉检查。不要把排版脚本写死为本 Skill 的运行依赖。

## 迭代规则

- 触达不够：先改传播入口、题目、场景或形式。
- 有播放但少主页访问：改用户情况与相关性。
- 有主页访问但少关注：改人物标签或内容承诺。
- 有认同但不相信能力：补过程、案例、比较和结果。
- 有咨询但不付款：先检查问题严重度、交付范围、价格、风险和买方条件。
- 有付款但结果弱：停止放大，先修产品和交付。
- 持续吸引错误人群：改“用户在什么情况下会想起你”和公开语言。
- 真实素材迅速枯竭：回到个人资产与 100 题检查。

## 不可违反的规则

- 不编造经历、数据、客户、市场需求、平台规律或付款。
- 不把外部案例、公开样本或搜索快照冒充本人账号结果。
- 不把旧产品的付款自动算作新定位或新产品已验证。
- 不用一个人的收入、家庭结构、平台、内容比例、产品价格或测试阈值作为通用标准。
- 不消费儿童、客户、伴侣或脆弱经历；涉及他人时取得授权并脱敏。
- 不用恐惧、羞辱、夸大结果或虚假稀缺推动成交。
- 不输出只有灵感清单、没有推荐决定和验证条件的战略书。
- 优先用普通话解释：写“用户在什么情况下会想起你”“成为主定位前必须通过的基础检查”“什么情况说明这条路不成立”；专业术语只能作为括号补充。
