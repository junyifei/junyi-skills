# Content and conversion system

## Start from the visitor's situation

Write one sentence:

> When [specific person] is in [trigger situation] and wants [change], this site should help them [next decision/action].

Separate:

- `reach audience`: people who may recognize and share the idea;
- `actual buyer/user`: people with the need, authority, and willingness to act;
- `non-fit`: people the offer should not attract;
- `entry temperature`: unfamiliar, aware, referred, or existing customer.

## Define the path

Use this sequence as a reasoning tool, not a mandatory page layout:

`entry context -> recognition -> useful change -> credibility -> offer or next step -> primary action`

The homepage may use different modules, but each module must answer a visitor question. Examples:

| Visitor question | Possible module |
|---|---|
| Is this for me? | first-screen promise and specific situation |
| Do they understand my reality? | scenes, costs, objections, or user language |
| What changes? | method, outcomes with boundaries, deliverables |
| Why trust this person? | process, artifacts, cases, results, relevant story |
| What exactly can I do next? | service, resource, booking, contact, or subscription |
| Is it safe and suitable? | fit/non-fit, privacy, FAQ, terms, limits |

## Calls to action

Choose one primary visitor action per page. Secondary utility actions may exist—for example language switch, navigation, or downloading an already-promised resource—but they must not compete with the main decision.

A CTA must state what happens next. Do not use false scarcity, hidden subscriptions, pre-checked consent, invented gifts, or a QR code that has not been tested.

## Evidence ledger

Track every trust element:

| ID | Public claim | Source | Date | Permission | Strength | Where used |
|---|---|---|---|---|---|---|
| P01 |  |  |  | confirmed/pending/no | fact/bounded claim/hypothesis |  |

Do not use a placeholder testimonial or logo in a public build. A gray card labeled “testimonial coming soon” can still imply evidence that does not exist; remove the module until proof exists.

## Copy review

For every section ask:

1. Which visitor question does this answer?
2. Is the claim supported at this strength?
3. Is it specific enough to recognize but broad enough to be true?
4. Does it repeat another section?
5. What should the visitor understand or do next?

Keep the creator's voice. AI may offer alternatives, but the user approves factual claims and final tone.
