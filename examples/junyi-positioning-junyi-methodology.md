# `junyi-positioning` 案例：君一方法论 GitHub 定位

这是一个真实的仓库定位案例。它展示决定过程，不证明改版后一定带来 Stars、安装或商业转化。

## 原始问题

仓库已经公开多个 Skills，但首页主要介绍产品结构、方法原则和目录。陌生用户仍难以回答：

- 这是为谁准备的；
- 我什么时候需要它；
- 它解决的共同问题是什么；
- 为什么这些不同能力属于同一个品牌。

## 可使用的事实

- 仓库有一个总路由和多个独立 Skill；
- 正式 Skill 要求真实任务、证据、边界和验收标准；
- 未验证的 IP Skills 已从公开仓库撤回；
- 君一的真实实践同时覆盖一人公司、内容、知识、家庭和 Agent 协作；
- 当前两个 IP Skills 的成熟度不同：定位已投入项目，对标仍在测试。

## 不能当成事实的内容

- 公开 Skill 已经帮助大量用户涨粉或成交；
- Stars、下载或关注一定会增长；
- 新的创始人经验共创服务已经完成支付和交付验证。

## 定位决定

### 首先服务的人

已经开始用 AI 做真实工作，但 AI 仍然接不住其经验和判断的一人公司经营者与知识型创作者；其中最鲜明的核心人群，是同时承担事业、家庭和自我成长的创业妈妈。

### 首先解决的问题

把真实材料、长期经验和判断标准变成 Agent 可以调用、可以验证、可以持续修订的方法；让重复执行交给 AI，把最终判断、责任、关系和重要生活留给人。

### 用户转述句

如果你已经在用 AI 做真实工作，却发现输出越多、审核越多，可以看看君一方法论。它公开展示真实经验怎样被蒸馏成 Agent 可调用的方法，也明确哪些判断不能交给 AI。

## 五层表达

| 层 | 决定 |
|---|---|
| 传播入口 | AI 越来越强，我为什么反而越来越忙 |
| 人物记忆 | 一人公司妈妈，带着一群 AI 在真实公司与生活里工作 |
| 内容承诺 | 公开 AI 怎样干活、翻车、被纠正，以及经验怎样变成 Skill |
| GitHub 产品 | 有证据、有边界、可调用、可验证的 Agent 方法库 |
| 商业落点 | 让 AI 开始学会创始人的经验和判断；当前仍处于首轮验证 |

## 本次最小修复

1. 把“服务谁、解决什么”放到 README 首屏；
2. 用真实处境替代抽象 Skill 列表作为第一层导航；
3. 提供可以直接复制的完整任务；
4. 增加安装、兼容性和调用语法说明；
5. 展示证据等级、成熟度和不承诺事项；
6. 建立问题、使用反馈和共创意向入口。

## 仍需验证

- 新首屏是否让目标用户更快判断相关性；
- 安装到第一次完成任务的流失点；
- 哪一个公开 Skill 最容易产生首次价值；
- 使用反馈能否形成可公开的修改证据；
- 共创意向是否来自真正有稳定业务和判断成本的人。
