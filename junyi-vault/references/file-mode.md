# 归档模式

## 归类判断顺序

1. 这条内容未来最可能为什么任务被找回来？
2. 当前结构中哪个域负责这个任务？
3. 哪个类对同类内容形成了稳定承诺？
4. 是独立档案，还是应追加到现有持续文件？
5. 是否需要在另一处放链接，而不是复制两份正文？

给出：首选位置、置信度、判断依据、第二候选及为什么不选。

## 置信度

- **高**：现有类定义和同类文件都支持，可以预览后直接归档。
- **中**：有一个首选，但另一类也合理；展示差异并让使用者确认。
- **低**：结构没有容纳位置、内容用途不明或可能涉及新域；不要写入。

## 文件策略

- 优先遵循现有文件名和 frontmatter。
- 没有约定时使用 `YYYY-MM-DD_一句话主题.md`。
- 默认一条独立事件/产物一个文件；持续人物、项目或长期看板可追加，但必须先读取现有结构并预览追加位置。
- 同名文件默认报错。使用者可以选择：取消、使用安全后缀、追加到已确认文件、明确覆盖。

## 默认最小 frontmatter

只有知识库没有现成约定时使用：

```yaml
---
date: YYYY-MM-DD
title: 一句话标题
source: 来源文件、链接或对话
tags: []
privacy: private
---
```

不要为了“结构化”增加无法稳定维护的字段。事件日期不明确时写 `date: unknown` 或询问，不用今天冒充事件日期。

## 批量归档

批量操作先生成完整 manifest 并一次性校验路径、冲突和字段。默认 dry-run，不允许前五条写入成功、后五条才发现路径错误。大批量应拆成可复核的小批次。
