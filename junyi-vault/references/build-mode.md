# 搭建模式

## 先问什么

优先从真实使用场景收集信息，不先让使用者发明抽象分类：

1. 未来一个月最常放进来的五类内容是什么？
2. 找资料时通常先想“属于生活哪一块”，还是先想“这是项目/人物/内容类型”？
3. 哪些内容需要长期维护，哪些只是临时收件？
4. 哪些资料含家庭、客户、健康、财务等敏感内容？
5. 当前使用 Obsidian、普通文件夹还是其他 Markdown 工具？

## 五类起手模型

这些只是提问起点，不是必须照抄的模板。

### 内容创作者

- 定位与品牌：定位、用户、表达边界
- 创作：选题、素材、草稿、已发布、复盘
- 学习：课程、书籍、行业观察
- 经营：产品、合作、数据

### 一人公司

- 业务：产品、营销、销售、交付、复盘
- 客户：客户档案、沟通、需求、案例
- 系统：SOP、工具、自动化、决策
- 个人：学习、健康、生活

### 家庭记录者

- 家庭：共同生活、关系、健康、财务
- 孩子成长：观察、里程碑、作品、教育
- 个人成长：学习、反思、兴趣
- 资料：待处理与公共参考

### 上班族

- 工作：项目、会议、协作、复盘
- 专业成长：课程、读书、技能、行业
- 生活：健康、财务、兴趣、家庭
- 资料：待处理与公共参考

### 已有混乱知识库

不直接套模板。先进入诊断模式，根据真实内容和检索习惯决定保留哪些域。

## 方案门槛

- 域通常 3–6 个；每个域内先保留 2–5 个会持续使用的类。
- 每个类写一句“收什么”和一句“不收什么”。
- 为三个典型输入演示归档位置；如果示例总要跨三四个分类，结构还不够清楚。
- 不预建大量空目录。暂不确定的内容进入一个明确的“待处理”入口，而不是创建“其他/杂项”永久黑洞。

## 初始化产物

确认后可创建：

- 域/类目录；
- 根目录 `知识库说明.md`：目的、原则、隐私和维护方法；
- 根目录 `索引.md`：域、类、用途与入口；
- 可选 `待处理/`：明确清理频率和退出条件。

操作 manifest 示例：

```json
[
  {"type": "directory", "path": "创作/素材"},
  {"type": "file", "path": "知识库说明.md", "content": "# 知识库说明\n"},
  {"type": "file", "path": "索引.md", "content": "# 索引\n"}
]
```
