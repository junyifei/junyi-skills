## 1. 三条证据线

### 注意力

回答：合适的人是否愿意停留、进入主页和继续关注？

可观察：完播/读完、分享、收藏、主页访问、关注、重复观看、目标用户评论。浏览或点赞只能证明低成本注意。

### 信任

回答：用户是否觉得被理解，并相信这个人有能力帮助产生变化？

分开记录：

- 被理解和喜欢：有共鸣的评论、真实故事反馈、价值观认同；
- 相信能力：具体追问、方法复述、过程理解、案例咨询、愿意提交真实场景。

### 支付

回答：合格买方是否愿意为明确结果承担真实成本？

强证据：付款、定金、复购、续费、转介绍、可核验结果。问价和报名意向只算前置信号。

## 2. 实验卡

每个实验发布或销售前填写：

| 字段 | 内容 |
|---|---|
| 编号 | H01 |
| 假设 | 哪类人遇到什么情况，会对什么主张做出什么行动 |
| 只改变的变量 | 入口/场景/内容承诺/证明/产品/价格等之一 |
| 刺激 | 内容、主页、案例、邀请、产品说明或交付 |
| 目标人群 | 触达人群或真正买方 |
| 主要指标 | 只选一个主要指标，其他作辅助 |
| 本人基线 | 同平台、同形式、相近账号阶段 |
| 最低样本 | 至少三次同类内容；支付按合格买方数 |
| 时间窗口 | 24 小时/7 天/30 天等 |
| 混杂因素 | 发布时间、形式、投放、热点、合作、平台波动 |
| 通过 | 预先定义 |
| 混合 | 哪些信号冲突或样本不足 |
| 不通过 | 预先定义 |
| 下一动作 | 三种结果分别怎么做 |

阈值优先来自本人历史同形式基线。没有基线时先发布一组基线内容；不要把其他账号绝对播放量当作及格线。触达不足时可判“样本不足”，不要自动判“需求不存在”。

## 3. 30 天验证

### 第 0-7 天：启动与基线

- 固定一个暂定方向和最多一个挑战方向；
- 统一人物标签、内容承诺和主页表达；
- 记录过去或新发布的同形式基线；
- 建立素材库、证据库、用户原话库和线索台账；
- 盘点 20 个真实现场、10 个能力证据和 3 个可交付小问题；
- 检查一周内容产能，不用无法持续的频率证明决心。

### 第 8-30 天：三线验证

- 每个关键注意力假设至少 3 次同类内容；
- 至少展示 3-4 个完整过程、失败或前后对比；
- 收集目标用户原话，区分泛共鸣与具体问题；
- 向合格买方提出一个真实收费或有成本承诺的最小实验；
- 每周只选择一个主要变量调整；
- 第 30 天按通过/混合/不通过判断，不用平均点赞宣布定位成功。

30 天的目标不是证明一个宏大定位永远正确，而是找到：一个较稳定的用户情况、一个能建立能力信任的内容机制、一个真实的付费场景。

## 4. 90 天路线

| 阶段 | 核心任务 | 最低产出 | 进入下一阶段的条件 |
|---|---|---|---|
| 0-7 天 | 资料、基线、表达与产能 | 证据账本、主页、首批内容 | 能稳定执行一周 |
| 8-30 天 | 注意力/信任/支付验证 | 受控内容、过程证据、最小付费实验 | 至少一个内容系列和一个付费场景出现正信号 |
| 31-60 天 | 放大内容赢家，修复交付 | 2-3 个案例、方法产物、第二场景 | 出现可公开结果与续费/转介绍信号 |
| 61-90 天 | 只放大赢家 | 内容/销售/交付的可复用流程 | 结果与产能允许扩大，而不是靠本人加班 |

阶段门必须从项目实际规模设置。案例数、价格和发布量是设计变量，不是通用真理。

## 5. 只修改失败层

| 观察 | 更可能出问题的层 | 先改什么 |
|---|---|---|
| 合适的人几乎没看到 | 传播入口或形式 | 场景、题目、开头、分发 |
| 有浏览，少主页访问 | 用户情况不够相关 | 把时刻、后果和对象说具体 |
| 有主页访问，少关注 | 人物标签或内容承诺 | 明确持续能看到什么 |
| 有共鸣，不相信能力 | 证明系统 | 过程、产物、案例、比较 |
| 有能力认可，无咨询 | 商业连接不清 | 买方问题和下一步 |
| 有咨询，无付款 | 产品或买方适配 | 问题严重度、范围、价格、风险 |
| 有付款，结果弱 | 交付 | 停止扩张，修产品 |
| 反复吸引错误人群 | 用户想起你的情况 | 改公开语言和筛选 |
| 写不出持续内容 | 个人资产或方向 | 回到真实材料和 100 题检查 |

只有同一结构问题在多次受控测试中重复出现，或本人发现伦理、身份、生活方式的重大不匹配，才考虑重建整个定位。

## 6. 语言检查

优先使用用户能直接听懂的表达：

| 优先写 | 可在括号中补充 |
|---|---|
| 用户在什么情况下会想起你 | 召回触发 |
| 成为主定位前必须通过的基础检查 | 硬门 |
| 什么情况说明这条路不成立 | 证伪条件 |
| 被理解和喜欢的信任 | 共情信任 |
| 相信你能交付结果的信任 | 能力信任 |
| 传播入口 | 大众入口/流量入口 |
| 商业落点 | 商业锚点 |
| 暂定方向 | 定位假设 |

发现抽象词时，补一个可观察翻译：

- “成长”→ 哪种行为或能力改变；
- “自由”→ 哪块时间、选择权或成本改变；
- “赋能”→ 具体完成什么；
- “高价值”→ 对谁、解决什么高成本问题；
- “生命智慧”→ 在哪种真实选择中体现；
- “系统化”→ 哪个流程、产物或结果可重复。

## 7. 质量与案例污染检查

### 事实纪律

- [ ] 每个关键结论有事实、推断、假设或未知标签。
- [ ] 数字有来源、日期、口径和适用范围。
- [ ] 外部市场样本没有冒充本人账号结果。
- [ ] 旧产品付款没有冒充新产品验证。
- [ ] 未知没有被顺滑文案掩盖。

### 方法完整性

- [ ] 写清用户在什么情况下会想起这个人。
- [ ] 区分五层定位。
- [ ] 区分触达人群、真正买方和不需要吸引的人。
- [ ] 用户变化包含可观察的之前和之后。
- [ ] 内容、故事、能力证明和产品都回到同一变化。
- [ ] 候选先做基础检查再评分。
- [ ] 30/90 天计划包含通过、混合、不通过和修改层。

### 通用性与案例污染

- [ ] 把姓名替换后，核心流程仍然成立。
- [ ] 没有把某人的收入、家庭身份、平台、内容比例、价格、产品名或案例数写成默认标准。
- [ ] 案例中的数字和判断只存在于案例段落，且标明“不可照抄”。
- [ ] 删除可选案例文件后，SKILL.md 与所有通用 references 仍能完整运行。
- [ ] 非案例输出没有出现案例人物姓名、产品名或专属口号。

### 可读性

- [ ] 开头先给决定，不让使用者读完全书才知道建议。
- [ ] 每个术语第一次出现时用普通话解释。
- [ ] 没有只写“召回触发、硬门、证伪”而缺少通俗说法。
- [ ] 每张表帮助比较或执行，不为显得专业而增加。
- [ ] 最后给出未来 7 天和 30 天动作。

### 伦理与执行

- [ ] 他人隐私、授权和脱敏边界清楚。
- [ ] 不承诺无法控制的收入、健康、关系或平台结果。
- [ ] 计划符合真实产能和生活方式。
- [ ] 不以焦虑、羞辱或虚假稀缺驱动传播和销售。

可运行 `scripts/check_strategy_book.py` 做基础结构和语言检查；机器通过不代表事实已核验，最终仍需逐条回到证据账本。
