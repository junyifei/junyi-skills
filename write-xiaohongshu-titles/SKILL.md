# 小红书封面与标题

把原始内容变成可被目标用户认领、可被正文兑现的小红书封面与标题。只借鉴外部案例的需求结构，不复制措辞、故事或人设。

## 读取上下文

1. 每次读取 [platform-method-and-cases.md](references/platform-method-and-cases.md)。
2. 用户提供定位、语气指南、禁用词或账号历史数据时，将其作为当前账号上下文，不假设 Skill 内置了某位创作者的人设。
3. 当存在当前账号的小红书发布数据或用户要求复盘时，读取 [calibration.md](references/calibration.md)。
4. 用户给出的原文、逐字稿和事实优先于所有参考资料；参考案例不能替代第一方证据。

## 输入不足时

优先从现有材料推断不改变事实的字段，并标记为 `推断`。只有缺失信息会改变标题真实性、人设或传播任务时才询问。不得补写不存在的经历、数字、对话、结果或身份。

最低输入为以下任一种：

- 一篇原文、逐字稿或口播稿；
- 一个有事实细节的真实事件；
- 包含目标用户、核心观点和证据的内容简报。

若只有抽象观点、没有事件或方法证据，仍可生成有边界的观点标题，但必须降低主张强度并说明缺口。

## 工作流

### 1. 建立证据账本

提取并编号：

- `事实`：身份、时间、动作、数字、结果；
- `原话`：可直接引用的原文句子；
- `事件`：发生了什么、预期与实际有什么反差；
- `判断`：创作者从事件中得到的有边界结论；
- `未知`：标题不能越过的空白。

### 2. 确定本条内容的唯一任务

从以下选择一个主任务：

- `被发现`：让陌生目标用户停下来；
- `搜索收藏`：回答一个具体问题，承诺可保存的方法；
- `建立人设`：让用户记住创作者可验证的身份、立场及其真实选择；
- `建立信任`：用过程、失败或证据证明能力；
- `促进需求`：让合格买方认出自己的问题，不做强推销。

### 3. 写清用户触发时刻与点击承诺

用一句用户自然语言回答：目标用户刚刚发生了什么，为什么此刻会点开。再用一句话说明点开后能得到什么。不要把“创作者想表达什么”误当成用户的点击理由。

### 4. 生成真正不同的角度

默认生成 5 套；证据较少时至少生成 3 套并说明原因。优先从不同角度中各取一种：

1. `处境镜像`：说出用户此刻的大白话；
2. `身份反差`：创作者两个或多个可验证身份之间的真实错位；
3. `事件结果`：具体任务、时间、数字、翻车或结果；
4. `认知反转`：原来以为是 A，事件证明更接近 B；
5. `方法收藏`：步骤、检查表、判断标准或可复用动作。

不得把同一个钩子换五种说法。每套都要写明 `angle`、`promise` 和所用 `source_ids`。

### 5. 分开封面与标题的工作

- 封面负责处境、情绪、反差或身份认领；写 2–4 行，每行编辑目标不超过 10 个非空白字符。
- 标题负责搜索词、故事缺口、方法、数字或收益；编辑目标不超过 20 个非空白字符。
- 封面与标题必须互补，不复述同一句信息。
- 默认不用 emoji；这是本 Skill 的编辑约束，不冒充平台硬规则，用户可以用自己的品牌规范覆盖。
- 标题必须像人说话，避免“干货整理、建议收藏、多数人都没想到、先看完再说”等空泛平台腔。

### 6. 评分与淘汰

每套按 100 分评分：

| 维度 | 分值 |
|---|---:|
| 原文证据与主张匹配 | 25 |
| 目标用户触发时刻 | 20 |
| 具体性与画面 | 15 |
| 点击承诺与信息增量 | 15 |
| 创作者定位与语气 | 15 |
| 封面标题互补 | 10 |

出现以下任一情况直接淘汰：编造事实；标题正文不一致；借别人身份或结果；消费孩子隐私；承诺轻松赚钱；用恐惧、羞耻或夸张结果逼点击；只是把外部爆款换同义词。

### 7. 输出并校验

先给简洁可读版，再附可校验 JSON。JSON 使用以下结构：

```json
{
  "platform": "xiaohongshu",
  "source_summary": "",
  "content_job": "被发现",
  "target_moment": "",
  "evidence": [
    {"id": "E1", "type": "事实", "text": ""}
  ],
  "options": [
    {
      "id": "X1",
      "angle": "处境镜像",
      "cover": ["行1", "行2"],
      "title": "标题",
      "title_chars": 0,
      "promise": "",
      "source_ids": ["E1"],
      "score": 0,
      "risk": ""
    }
  ],
  "recommended_id": "X1",
  "test_hypothesis": ""
}
```

把 JSON 保存为文件后运行：

```bash
python3 scripts/validate_titles.py result.json
```

若有原文文本文件，再运行：

```bash
python3 scripts/validate_titles.py result.json --source source.txt
```

修正所有错误后再交付。警告项需要人工判断并在 `risk` 中说明。

## 数据更新边界

当前外部案例只构成首版假设，不代表小红书算法规律。不得声称固定权重、万能公式或保证爆款。当前账号积累至少 12 条可比较的小红书数据后，按 [calibration.md](references/calibration.md) 更新角度优先级，不因单条爆款改写方法。
